ブラウザだけで、Word と同じ見た目を。

Oxi は、Word 文書（.docx）をブラウザ上で Word そのままのレイアウトで表示・編集するためのオープンソースエンジンです。Rust + WebAssembly でゼロから実装し、再現度は実在の文書で Microsoft Word とページごとに照合。サーバーは使わず、ファイルは 1 バイトも外に出ません。

再現度は「主張」ではなく「実測」

ここに並ぶ数値はすべて、リポジトリで公開している測定パイプラインで誰でも再現できます。

日本語コーパス — 235 文書・410 ページ

就業規則、官公庁の申請様式、契約書……実在の日本語ビジネス文書 235 件（410 ページ）を Microsoft Word の出力と重ね、ピクセル単位の SSIM（構造的類似度）で採点しています。2026-07-17 時点の成績はページ平均 0.9378、文書平均 0.9591。満点（= Word と完全一致）は 1.00 です。

日本語ブラインドセット — 初見の 50 文書

上の 235 文書コーパスは開発にも使っているため、それを補う凍結ブラインドベンチマークを用意しました。公開 docx-corpus の日本語 50 文書を、測定する前の 2026-07-20 に凍結し、修正のターゲットには一切使いません — すべてのスコアが「初見」の汎化性能です。初回測定（2026-07-21）で Oxi は文書平均 0.828。同じ文書での LibreOffice（0.816）を上回り、50 文書中 33 文書で Word により近い結果でした。ページ数は 44/50 で Word と一致。別枠の段落単位ページネーション検証では、測定可能な 48 文書のうち 41 文書で、対応づいた全段落が Word と同じページに配置されました。グラウンドトゥルースは Microsoft Word、比較は LibreOffice 26.2.1.2、いずれも 150 DPI で描画しています。

英語コーパス — 別枠で測って、そのまま公開

英語は最初から一番厳しい測り方をしています — 公開 docx-corpus から凍結したブラインドベンチマーク 50 文書。修正対象には一切使わないので、出てくる数字はすべて「初見の実力」で、テストに合わせた調整が入り込む余地がありません。2026-07-20 時点で文書平均 Oxi 0.807（初回測定 2026-07-19 は 0.800）。ONLYOFFICE（0.902）と LibreOffice（0.876）にはまだ及びませんが、アーキテクチャが最も近い競合 — Rust + WebAssembly で Canvas に描く SILURUS（0.776）— は上回っており、50 文書中 32 文書で Oxi が勝っています。残る差が英語側の作業キューです — エンジンの改善に合わせて再測定し、上のグラフの線を伸ばしていきます。比較対象: 正解は Microsoft Word（Microsoft 365 16.0.20131.20154）、ONLYOFFICE 9.3.1.8、LibreOffice 26.2.1.2、SILURUS @silurus/ooxml 0.72.2 — すべて同日に 150 DPI でレンダリング。

なぜこの精度が出せるのか

  • 100% クリーンルーム実装：逆アセンブルやバイナリ解析は一切行いません。仕様の根拠は公開標準（OOXML — ISO/IEC 29500 / ECMA-376）と、COM API 経由で Word の出力値を観測するブラックボックス測定の 2 つだけです。
  • デュアルフォントエンジン：Word のレイアウトは GDI の整数ピクセル丸めを前提に組まれているため、.docx では GDI の振る舞いをそのまま再現。そのしがらみがない形式には DirectWrite の浮動小数メトリクスを使います。
  • 回帰ゲート：すべての変更を全コーパスで検証。改ページの一致、要素の位置、ピクセル SSIM を毎回確認してから取り込みます。

日本語組版を、中心に

多くの Word 互換レンダラーで後回しにされてきた日本語レイアウトこそ、Oxi の実測の中心です。測定コーパス自体が、日本語レイアウトが最も難しい文書群——省庁の規程、行政の申請様式、雇用契約書——でできています。

  • 禁則処理 — JIS X 4051 に沿った行分割。行頭・行末禁則、約物の詰め、ぶら下がり組み。
  • 縦書き — 右から左に流れる縦組み。縦中横にも対応。
  • ルビ（ふりがな）、割注、圏点（傍点）。
  • 文字グリッド（docGrid） — Word の行グリッド・字送りモデルを忠実に再現。
  • 判子 — 丸印・角印・小判型の電子印鑑生成と PAdES PDF 署名。

技術

Rust + WebAssembly、形式に依存しない中間表現、そしてサーバーゼロ。

なぜ今なのか

2026 年、フランス（DINUM 指令）、スイス（連邦事務局）、ドイツ（OpenDesk）など、欧州の公共部門は Microsoft Office 離れを本格化させています。そしてどの移行でも、最後に残るのは同じ問題です — 手元にある大量の .docx が、Word と同じ見た目で開けるかどうか。Oxi は Word 再現度を「約束」ではなく「継続的な実測」で示すことで、この問題そのものをなくすことを目指しています。

アーキテクチャ

  • 完全クライアントサイド：解析・レイアウト・描画・編集はブラウザ内の WebAssembly で実行。サーバー基盤もアップロードも不要です。
  • 形式に依存しない中間表現：Document → Page → Block (Paragraph | Table | Image) → Run。OOXML にも ODF にも寄せていないので、往復変換で劣化しません。
  • ラウンドトリップ編集：元の ZIP を保持し、変更した XML テキストノードだけをパッチ。ファイルをゼロから作り直すことはしません。
  • 同梱フォントはオープンライセンスのみ：メトリクス互換のオープンフォントを使用し、プロプライエタリフォントは同梱しません。
  • 組み込み可能：wasm-bindgen の JavaScript API（parse_document、layout_document、edit_docx など）を任意の Web アプリから呼び出せます。

ライセンス

コアエンジンは MPL-2.0（ファイル単位コピーレフト。AGPL と違い、商用製品への組み込みも可能です）。WebAssembly / Python バインディングは MIT OR Apache-2.0。コントリビューションは DCO のみで CLA はありません。

よくある質問

無料で使えますか？

はい。オープンソース（コア MPL-2.0、バインディング MIT OR Apache-2.0）で、Web デモは登録不要・無料です。

ファイルはサーバーに送信されますか？

されません。処理はすべてブラウザ内の WebAssembly で完結し、ファイルが外部に送られることはありません。機密文書でも安心して使えます。

Word とどれくらい同じ見た目になりますか？

日本語の実文書 235 件（410 ページ）で Word とピクセル単位の SSIM を継続測定しており、2026-07-17 時点でページ平均 0.9378 / 文書平均 0.9591（英語のブラインド 50 文書は文書平均 0.807 — アーキテクチャが最も近い SILURUS を上回り、LibreOffice・ONLYOFFICE にはまだ及びません）。同じ条件で LibreOffice・ONLYOFFICE を上回ります。比較グラフと測定方法は再現度セクションをご覧ください。

対応している形式は？

まずは Word の .docx です。ページレイアウトの完全再現とラウンドトリップ編集に集中しています。「Word の .docx 描画を忠実に再現する」ことが Oxi のミッションで、対応形式を広げるのはその先です。

自社製品に組み込めますか？

組み込めます。コアはファイル単位コピーレフトの MPL-2.0、バインディングは MIT OR Apache-2.0 なので、AGPL 系と違い商用・プロプライエタリ製品にも組み込めます。

オフラインでも動きますか？

ページと WebAssembly モジュールを読み込んだあとは、文書の表示も編集もネットワークなしで動きます。
